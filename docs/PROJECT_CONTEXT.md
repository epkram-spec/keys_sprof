# PROJECT_CONTEXT.md

## Назва

Назва в інтерфейсі: Радар кейсів SPROF.
Технічна назва: SPROF Case Radar.

## Межі продукту

Радар кейсів SPROF - внутрішній вебсервіс для збору, оцінки, ведення і передачі потенційних кейсів у маркетинг.

Це не CRM, не система продажів, не таск-менеджер, не склад і не контент-план.

Не додавати без прямої інструкції:
- CRM-функції;
- продажі;
- склад;
- контент-план;
- AI-генерацію текстів;
- складний календар зйомок;
- задачі для команди.

## Сповіщення

Усі сповіщення мають відображатись у кабінеті.

In-app запис створюється завжди. Email і Telegram є тільки додатковими каналами доставки.

У хедері має бути дзвіночок із кількістю нових, не переглянутих сповіщень.

Кожен користувач бачить у дзвіночку тільки свої сповіщення згідно з роллю і правами доступу.

Якщо Telegram не привʼязаний, це не блокує сповіщення: лишаються in-app і email.

Однакові нагадування не дублювати. Для цього використовувати dedupe_key.

## Архітектура

Стек: Next.js App Router, TypeScript, Tailwind CSS, shadcn/ui, Supabase Auth, Supabase PostgreSQL, Supabase RLS, Supabase Storage, Vercel, Google Sheets API, email notifications, Telegram Bot notifications.

Supabase використовується для авторизації, бази даних, RLS і файлів.

Next.js відповідає за UI, серверні дії, маршрути API та інтеграції.

Google Sheets API використовується для імпорту даних із таблиць.

Email і Telegram використовуються тільки як додаткові канали доставки сповіщень, не як основне джерело стану.

## GitHub / DB / Vercel

Працювати по Gate-етапах в окремих гілках.

Агент не пушить у GitHub без команди власника: "Погоджую Gate X. Коміть і пуш".

Усі зміни БД робити тільки через файли міграцій у supabase/migrations.

Агент не застосовує міграції на staging або production без окремого погодження.

Агент не запускає Vercel staging/preview без окремого погодження.

Production deploy тільки після окремого підтвердження.

Секрети не комітити.

## Після кожного Gate-етапу

Запустити lint, typecheck і build, якщо відповідні команди вже є в проєкті.

Дати звіт:
- що зроблено;
- які файли змінено;
- чи є міграції;
- як перевірити;
- що не зроблено;
- які ризики лишились;
- чи потрібен GitHub push;
- чи потрібен Supabase staging;
- чи потрібен Vercel staging.

Після звіту зупинитись і чекати перевірки власника.
