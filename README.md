# Радар кейсів SPROF

Внутрішній вебсервіс для збору, оцінки, ведення і передачі потенційних кейсів у маркетинг.

Це не CRM, не система продажів, не таск-менеджер, не склад і не контент-план.

## Стек

- Next.js App Router
- TypeScript
- Tailwind CSS
- shadcn/ui-сумісна структура компонентів
- ESLint
- Prettier
- Zod для перевірки змінних середовища

## Локальний запуск

```powershell
npm install
npm run dev
```

Застосунок відкривається за адресою `http://localhost:3000`.

## Змінні середовища

1. Створи локальний файл `.env` на основі `.env.example`.
2. Заповни приватні значення тільки в `.env` або у Vercel Environment Variables.
3. Не коміть `.env` і секрети.

## База даних

Усі зміни БД робляться тільки через файли міграцій у `supabase/migrations`.

Міграції не застосовуються на staging або production без окремого погодження власника.

## Тестові користувачі

Реєстрація з інтерфейсу закрита. Тестових користувачів створює адміністратор у Supabase:

1. Відкрий Supabase Dashboard.
2. Перейди в Authentication -> Users.
3. Натисни Add user -> Create new user.
4. Вкажи email і пароль.
5. Після створення користувача перевір таблицю `profiles`.
6. Для першого адміністратора використовуй `epkram@gmail.com`.
7. Для інших користувачів вистав одну з дозволених ролей: `manager`, `marketing`, `leader`, `admin`.

Якщо профіль не створився автоматично, перевір, що застосована міграція Gate 2 з trigger `on_auth_user_created`.

## Деплой і GitHub

Агент не пушить у GitHub, не застосовує міграції і не деплоїть на Vercel без явної команди власника.

## Підготовка до staging

Перед staging потрібно:

1. Переконатися, що у Vercel задані всі змінні з `.env.example`.
2. Окремо додати приватні секрети у Vercel Environment Variables, не в GitHub.
3. Застосувати Supabase міграції тільки після окремого погодження власника.
4. Перевірити першого адміністратора: `epkram@gmail.com`.
5. Створити тестових користувачів через Supabase Authentication.
6. Перевірити ролі: менеджер, маркетинг, керівник, адміністратор.
7. Перевірити, що Storage bucket `case-files` приватний.
8. Перевірити Google Sheets service account і доступ до таблиць.
9. Перевірити SMTP і Telegram env, якщо потрібна зовнішня доставка сповіщень.

## Staging smoke test

1. Увійти адміністратором.
2. Створити менеджера, маркетинг і керівника через Supabase.
3. Менеджером створити кейс, додати файл і коментар.
4. Перевірити автоматичний скоринг і пояснення балів.
5. Передати кейс у маркетинг.
6. Маркетингом змінити статус у панелі маркетингу.
7. Перевірити дзвіночок, read/unread і сторінку сповіщень.
8. Перевірити імпорт Google Sheets через preview і ручне підтвердження.
9. Керівником або адміном завантажити CSV.
10. Адміном перевірити журнали й довідники.
